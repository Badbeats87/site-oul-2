Usage Guidelines

Label Best Practices
  • Always provide descriptive labels for form inputs
  • Use labels, not just placeholders (placeholders disappear when typing)
  • Keep labels concise but clear
  • Indicate required fields with * or "required"
Validation
  • Show error messages immediately below the input
  • Use .form-error class for error messages
  • Use .form-success class for success messages
  • Maintain error message text until user corrects and resubmits
Help Text
  • Use .form-help for additional guidance
  • Keep help text short and informative
  • Explain requirements, formats, or constraints
  • Not for error messages or validation feedback
Input Types
  • text: General text input
  • email: Email validation enabled
  • tel: Telephone number input
  • number: Numeric input with spinners
  • date: Date picker (browser-dependent)
  • time: Time picker (browser-dependent)
Accessibility Checklist:
  • ✓ All inputs have associated labels (use for/id)
  • ✓ Error messages linked via aria-describedby
  • ✓ Required fields marked and announced
  • ✓ Keyboard navigation works (Tab, Shift+Tab)
  • ✓ Focus visible on all interactive elements
  • ✓ Help text programmatically associated
  • ✓ Success/error states conveyed via color AND text/icon
